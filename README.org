#+TITLE: CAN-J1939
#+AUTHOR: Rickard Häll

A J1939 sensor node for Linux, built on top of the kernel J1939 socket stack.
The node claims an address dynamically on startup, responds to identity requests
from other nodes on the bus, and is designed to be extended with real sensor
hardware.

** Documentation
- Kernel J1939 socket documentation: https://www.kernel.org/doc/Documentation/networking/j1939.rst
- can-utils kickstart guide: https://github.com/linux-can/can-utils/blob/master/can-j1939-kickstart.md

** Architecture

Three threads operate concurrently over a single J1939 socket:

- ~rx_thread~ — receives all incoming frames. Handles network management
  (address contention, commanded address) directly and queues on-request PGN
  responses for TX.
- ~tx_thread~ — drains the on-request queue and sends PGN response payloads.
  Also broadcasts periodic sensor PGNs driven by ~pgn_tasks[]~.
- ~sensor_thread~ — polls hardware sensors at configurable intervals and
  writes values into a shared ~sensor_values_t~ struct protected by a mutex.

** Network Behavior

*** Address Claiming
On startup the node attempts dynamic address claiming starting at ~0x80~,
scanning upward through ~0xF7~ until an unchallenged address is found.
If the full range is exhausted a Cannot Claim Address frame is broadcast
and the node shuts down.

*** Contention Handling
The RX thread monitors Address Claimed frames (~PGN 60928~). If another node
claims our current address with a lower NAME value we yield and claim the next
available address upward. If no address is available the node shuts down.

*** Commanded Address
If a Commanded Address frame (~PGN 65240~) is received targeting our NAME
we attempt to claim the commanded address. If that address is contested we
fall back to dynamic claiming from ~PREFERRED_ADDRESS~. If that also fails
the node shuts down.

*** Supported PGNs (on request)

| PGN   | Name                     | Description                           |
|-------+--------------------------+---------------------------------------|
| 65259 | Component Identification | Make, Model, Serial, Unit             |
| 65242 | Software Identification  | Firmware version string               |
| 64965 | ECU Identification       | Part number, Serial, Location, Type   |

Any request for an unsupported PGN receives a NACK (~PGN 59392~) in response.

** Prerequisites

Ensure your CAN interface is up and the J1939 stack is loaded.

#+BEGIN_SRC bash
  # Load the J1939 module
  sudo modprobe can-j1939

  # Bring up a virtual CAN interface (for testing)
  sudo ip link add dev vcan0 type vcan
  sudo ip link set vcan0 up
#+END_SRC

** Building

#+BEGIN_SRC bash
  make
#+END_SRC

** Extending — Adding a Sensor

*** 1. Add the value to ~sensor_values_t~ in ~sensors.h~
#+BEGIN_SRC c
  typedef struct {
      float temperature; /* degrees Celsius */
  } sensor_values_t;
#+END_SRC

*** 2. Implement read and write functions in ~sensors.h~ and ~sensors.c~

The ~read~ function reads raw bytes from hardware into a caller-supplied buffer.
No lock is held during this call.

#+BEGIN_SRC c
  int sensor_temperature_read(void* buf);
#+END_SRC

The ~write~ function knows the type of the raw buffer and writes the value into
the correct field of ~sensor_values_t~. This is called under the sensors mutex.

#+BEGIN_SRC c
  int sensor_temperature_write(sensor_values_t* sensors, const void* buf);
#+END_SRC

Example implementations in ~sensors.c~:

#+BEGIN_SRC c
  int sensor_temperature_read(void* buf) {
      float value = /* read from hardware */;
      memcpy(buf, &value, sizeof(float));
      return 0;
  }

  int sensor_temperature_write(sensor_values_t* sensors, const void* buf) {
      memcpy(&sensors->temperature, buf, sizeof(float));
      return 0;
  }
#+END_SRC

*** 3. Register the task in ~ecu.c~
#+BEGIN_SRC c
  static sensor_task_t sensor_tasks[] = {
      {
          .poll_rate_ms = 1000,
          .read         = sensor_temperature_read,
          .write        = sensor_temperature_write,
      },
  };
#+END_SRC

*** 4. Add a PGN builder in ~pgn_data.c~
Add a new ~build_pgn_XXXXX_payload()~ function, register it in
~build_payload()~ and ~handle_request()~, and define the PGN number
in ~pgn_data.h~.

** Extending — Adding a Periodic PGN

Periodic PGN broadcasts are driven by ~pgn_tasks[]~ in ~ecu.c~. The TX thread
loops over the table every 10ms, checks elapsed time against each entry's
~tx_rate_ms~, snapshots ~sensor_values_t~ under the mutex, then calls
~build_payload()~ and ~can_send()~ — no per-PGN logic in the thread itself.

Add a PGN builder in ~pgn_data.c~, register it in ~build_payload()~, and add
an entry to ~pgn_tasks[]~:

#+BEGIN_SRC c
  static pgn_task_t pgn_tasks[] = {
      { .pgn = PGN_XXXXX, .tx_rate_ms = 1000 },
  };
#+END_SRC
